# 交易所连接说明

这份文档专门说明本项目里 `ccxt + Binance/OKX` 的正确连接方式，以及哪些写法会把图表加载链路带坏。

## 正确的连接方式

### 1. 公共行情请求和私有账户请求必须分开

图表加载、K 线获取、`exchangeInfo/load_markets` 这类行为属于公共请求，不应该依赖 API Key。

正确做法：

- 图表加载走 `create_exchange(..., require_auth=False)`
- 这类实例不注入 `apiKey/secret/password`
- 只用于：
  - `load_markets()`
  - `fetch_ohlcv()`
  - 公共市场信息查询

私有账户请求，比如：

- `fetch_my_trades()`
- `fetch_orders()`
- 提现/资金/账户类接口

必须显式走：

- `create_exchange(..., require_auth=True)`

这样职责才是清晰的。

### 2. 代理只能作为“可选网络层”，不能成为图表加载的硬依赖

当前环境变量支持：

```env
EXCHANGE_PROXY_URL=socks5://127.0.0.1:7890
```

正确做法：

- 如果代理可用，优先走代理
- 如果代理不可用，公共行情请求应该允许回退直连
- 如果你本机根本不用代理，直接把它留空：

```env
EXCHANGE_PROXY_URL=
```

### 3. 只看本地 CSV 时，不应该被账户配置卡住

如果你的目标只是：

- 读取 `data/*.csv`
- 选交易对
- 拉公共 K 线做复盘

那么即使 `.env` 里没有 API Key，也应该能正常加载图表。

## 错误的连接方式

### 错误方式 1：`require_auth=False` 但仍然偷偷注入 API Key

这是本次问题的关键根因之一。

错误写法的后果：

- 代码以为自己在做“公共图表加载”
- 但 `ccxt` 检测到 credentials 存在
- `exchange.load_markets()` 会先触发 `fetch_currencies()`
- Binance 的 `fetch_currencies()` 会走私有 `sapi`

典型报错就是：

```text
https://api.binance.com/sapi/v1/capital/config/getall
```

这说明图表加载链路被错误带到了私有接口。

### 错误方式 2：公共请求强依赖一个坏代理

如果 `.env` 配了：

```env
EXCHANGE_PROXY_URL=socks5://127.0.0.1:7890
```

但本机这个代理没开、地址写错、DNS 异常，`ccxt/requests/urllib3` 会先在 SOCKS 层失败。

典型报错特征：

- `socks.GeneralProxyError`
- `SOCKSHTTPSConnection`
- `Failed to establish a new connection`
- `nodename nor servname provided, or not known`

这类报错优先说明：

- 代理本身不可用
- 还没有正常进入交易所接口逻辑

### 错误方式 3：把“能不能访问交易所”与“能不能使用图表”绑死

本项目的复盘模式有两种：

1. 本地 CSV + 公共 K 线
2. 私有成交抓取 + 仓位重建

错误设计是把这两件事耦合在一起，导致：

- 只是看图表
- 也必须先把账户、代理、私有接口全部打通

这会让排障非常混乱。

## 本次问题的实际结论

这次 `/api/chart/load` 的 500 不是单一原因，而是两层叠加：

### 第一层：代理有问题

当前环境里：

```env
EXCHANGE_PROXY_URL=socks5://127.0.0.1:7890
```

而 traceback 最底层首先出现的是 SOCKS 连接失败，所以“当前请求为什么死掉”的直接原因是代理不可用。

### 第二层：代码把公共图表加载错误带进了私有接口

如果公共实例也带了币安密钥，那么：

- `load_markets()`
- `fetch_currencies()`
- `sapi/v1/capital/config/getall`

这条链路就会被触发。

这不是你想要的行为，因为图表加载本来只需要公共市场数据。

## 当前应该遵守的规则

后端里后续新增交易所逻辑时，按下面规则走：

### 公共请求

- 用于图表
- 用于 K 线
- 用于市场元数据
- `require_auth=False`
- 不注入密钥
- 允许代理失败后回退直连

### 私有请求

- 用于账户成交
- 用于订单
- 用于仓位重建
- `require_auth=True`
- 缺密钥直接报配置错误
- 是否走代理由运行环境决定

## 出现报错时怎么判断是哪一层

### 如果看到这些关键词

- `socks.GeneralProxyError`
- `SOCKSHTTPSConnection`
- `nodename nor servname provided`

优先判断：代理问题。

### 如果看到这些关键词

- `sapi/v1/capital/config/getall`
- `fetch_currencies`
- `load_markets`

优先判断：公共请求错误带上了私有 credentials。

### 如果看到这些关键词

- `api/v3/exchangeInfo`
- `fapi/v1/exchangeInfo`

一般说明还在公共市场信息阶段，先看代理和网络，再看 symbol/market type。

## 推荐排障顺序

1. 先确认 `.env` 里的 `EXCHANGE_PROXY_URL` 是否真的可用。
2. 如果本机不需要代理，直接清空这一项。
3. 再确认公共请求实例是否带了 API Key。
4. 最后再看 symbol、market type 和时间区间本身。

## 最小可用建议

如果你当前只是本机复盘，建议优先使用：

```env
EXCHANGE_PROXY_URL=
```

并保持：

- 图表链路走公共接口
- 仓位重建链路再显式用私有接口

这是当前这个项目最稳定的连接方式。
