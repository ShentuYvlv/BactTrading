# Variational 仓位重建修正说明

## 结论

旧版 `scripts/convert_variational_exports.py` 的核心错误，是把 `buy/sell` 直接当成“开多/开空”方向，再用净头寸方式去推断开平仓。  
这不符合 Variational 导出的实际语义。

这批导出里，真正可靠的口径是：

1. `30天PNL.xlsx` 里只要出现某个 `symbol + time` 的 realized pnl 事件，不管数值是否为空，这一秒对应的成交就是一次平仓事件。
2. `30天订单数据.txt` 里的 `Yes / No` 列可以作为开平仓辅助标签：
   - `Yes` 更接近平仓
   - `No` 更接近开仓
3. `30天PNL.xlsx` 的数值列并不完整：
   - 负收益基本都有值
   - 正收益大量缺失，导出后变成空值

所以正确重建逻辑应该是：

- 先识别“哪笔成交是平仓”
- 再按成交加权均价计算每次 realized pnl
- 如果 PNL 文件里该次平仓有数值，就以它为准
- 如果 PNL 文件里该次平仓数值缺失，就退回成交均价计算

## 旧逻辑为什么会错

以 2026-02-18 的 ETH / SOL 样本为例：

- `ETH-PERP 2026-02-18 21:17:35 buy 6.86856`
- `SOL-PERP 2026-02-18 21:17:35 buy 112.296`

旧逻辑把这两笔 `buy` 都当成“开多”。  
但站内真实含义是“买入平空”，因为同一时刻存在 realized pnl 事件，而且订单导出里这一行是 `Yes`。

因此旧逻辑会把：

- 之前连续卖出的空头仓位错误延长
- 或者把平仓错并进新开多仓
- 最终导致 position PnL 和站内 realized pnl 对不上

## 新逻辑

新版脚本采用了下面的优先级：

1. `PNL 事件存在` -> 该笔成交判定为平仓
2. 如果订单导出高置信度匹配到该笔成交：
   - `Yes` -> 平仓
   - `No` -> 开仓
3. 只有在没有任何提示时，才退回旧的净头寸兜底逻辑

每个仓位内部采用：

- 多次开仓加权均价
- 多次部分平仓逐笔结算
- 每次平仓优先使用 PNL 文件中的数值
- 缺失的正收益 PnL 用成交均价补齐

## 仍然无法 100% 还原的情况

有一类仓位只能“部分恢复”：

- 这笔交易在当前 30 天窗口内是平仓
- 但对应开仓发生在窗口之前
- 同时该次 realized pnl 在 `30天PNL.xlsx` 里又恰好是缺失的正收益

这种情况下，源文件里既没有开仓成交，也没有 PnL 数值，无法从这三份导出中唯一确定真实开仓价。  
脚本现在会生成一个“合成开仓”来保留这次平仓事件，但对应 PnL 只能保守处理。

已知样本：

- `ETH-PERP 2026-02-18 00:35:51`

站内截图显示它有正收益，但导出的 `30天PNL.xlsx` 该行是空值，因此无法仅靠本地三份导出完全恢复。
